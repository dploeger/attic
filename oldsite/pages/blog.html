First success with SloxML

I just had my first brilliant experience with sloxml: I used the Java GUI-Plugin to sync my contact database and it worked. I only returned the First- and LastName-Fields, but this worked.

Now I've added other fields to the output, but I have to understand the different state-methods of the SyncSource to go on. For now I've only developed the “GetAllSyncItems”-method for a slow sync (where the client has no data at all). I still have to go for “GetNewSyncItems”, “GetUpdatedSyncItems” and “GetDeletedSyncItems”.

The problem is, that I don't quite understand the concept behind these and what data I have to manage (do I have to record each item I have send to a principal or does the principal keep this database?), but I will investigate on that.

The good thing to notice is: it works!

SQLTools 1.1.1 released

That was fast. SQLTools Version 1.0.1 lasted for only one day. I've updated the SQLTable-object to specify a table join-type. The default (inner) can now be overwritten with “outer left” for example. This is specified using an expanded version of the SetJoin method. Read all about it in the API-documentation.

SQLTools 1.0.1 released

During the described development for sloxml I've come across the task to dynamically create SQL statements. After some googling I haven't found any package that allows the developer to create SQL statements object oriented.

So I've developed my own.

Yes. In Java.

I don't know what came over me.

Anyways, I've released a tool named “SQLBuilder” in a package I call “SQLTools”. This tool allows you to build up your SQL-Statement using objects and then building a real SQL statement out of ArrayLists of objects.

For more information go here: About.
